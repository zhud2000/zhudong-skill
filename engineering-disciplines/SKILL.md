# 工程纪律

## 概述

把普世开发原则转化为每次改动时自动执行的检查。不是建议，是纪律。

## 规则 1：改之前搜，改之后查（依赖检查）

**触发条件**：修改任何函数签名、字典键名、API 参数、Prompt 变量名

**执行步骤**：
1. `grep` 搜索旧名称在项目中的所有出现
2. 列出每一个引用位置
3. 逐条确认：需要同步修改？还是有意保留？
4. 全部改完后，再搜一次确认无遗漏

**不做的后果**：改了 Prompt 键名 → doc_assembler 还在读旧键名 → 章节空白。历史已证明。

## 规则 2：大改动前，先出影响分析

**触发条件**：改动预计影响 3 个以上文件

**执行步骤**：
1. 在改动前，先输出一句："这个改动会影响：[文件1] [文件2] [文件3]，因为 [原因]"
2. 列出每个受影响文件中需要改的具体位置
3. 改完之后逐条核对

**为什么**：人脑工作记忆有限。写出"全景图"可以释放工作记忆，让你专注于改动本身。

## 规则 3：每个改动带注释

**触发条件**：任何对现有代码的修改

**执行步骤**：
```python
# [YYYY-MM-DD] 为什么改：xxx。关联影响：需同步改 A、B。
```

**必须包含三个要素**：
- 日期
- 原因（为什么改，不是改了什么）
- 关联影响（改这个会影响哪些其他地方）

**为什么**：三个月后你不记得今天为什么这样写。注释是你留给未来自己的信。

## 规则 4：引入新工具前，先评估新问题

**触发条件**：决定使用新的第三方库、框架、API 或工具

**执行步骤**：
1. 列出这个工具解决的 3 个问题
2. 列出这个工具可能带来的 3 个新问题
3. 确认：新问题你有能力调试和解决

**真实案例**：Docker 解决了环境一致性，带来了缓存失效、网络隔离和构建延迟。这些新问题花掉了我们数小时。

## 规则 5：运行端到端之前不算完成

**触发条件**：任何修改后认为"做完了"

**执行步骤**：
1. 前端改了 → 检查 `npm run build` 是否通过
2. 后端改了 → 检查 `docker compose up -d` 是否成功
3. 数据格式改了 → 跑 `e2e_test.py`
4. 全部通过才算完成

**为什么**：代码在编辑器里正确 ≠ 系统在运行时正确。中间的差距是部署、编译、网络、缓存——每一个都曾经让我们踩坑。

## 不可机器化的（但同样重要）

以下原则无法写成可执行规则，但应该在每次做决策时自我提醒：

- **抽象是双刃剑**：每引入一层抽象，同时引入一层调试难度
- **保护注意力**：减少上下文切换。一个完整的小时 > 四个 15 分钟的碎片
- **穿越不确定**：真正的成长发生在"不知道能不能行"的阶段
