# Task Orchestrator

## Overview

防止长会话中执行质量下降。核心理念：**花 30 秒规划，省 30 分钟返工。**

不追求流程完整——追求的是一次做对。

## 触发判断（每次任务开始前自问）

```
这个任务有多复杂？
    │
    简单（<5步，单一主题）  →  跳过本 SKILL，直接做
    中等（5-10步）          →  精简模式：只做 Phase 0 + Phase 3
    复杂（>10步或跨系统）   →  完整模式：Phase 0→1→2→3
    会话已>15轮             →  不管任务大小，至少走精简模式
```

**精简模式 = 写 Todo + 最后自检。** 不要中间 Gate。不要 Red Flags 审查。做完检查就结束。

## 完整模式：4 个 Phase

```
Phase 0: 规划 (30秒)    →  TodoWrite，每项写清"怎么算完成"
Phase 1: 理解           →  读关键文件，确认需求边界
Phase 2: 执行           →  逐项做，做完一项验证一项
Phase 3: 自检           →  回答 3 个问题，全通过 = 结束
```

### Phase 0: 规划

调用 TodoWrite。每个 Todo 回答：**"做完这个，我怎么知道它真的做完了？"**

最少 Todo 数参考：
- Bug 修复: 3（复现→定位→修复→验证）
- 新功能: 4（理解→设计→实现→测试）
- 调研: 3（搜索→验证→综合）

### Phase 1-2: 执行

Phase 1 和 Phase 2 之间不需要显式 Gate。唯一规则：**发现理解有误就回到 Phase 0 改 Todo。**

工具调用失败时的处理：
- 第 1 次失败：换个方式重试
- 第 2 次失败：报告给用户，不要静默跳过
- 不允许："这个做不了，算了"（不告知用户就放弃）

#### 写代码时

**简洁优先（Simplicity First）** — 最少代码解决问题，不加任何预测性东西：
- 不添加用户没要求的功能
- 不为单一用途的代码建抽象层
- 不做"以后可能需要"的灵活性和可配置性
- 不对不可能发生的场景做错误处理
- 写了 200 行后发现 50 行能搞定 → 重写成 50 行

**手术式修改（Surgical Changes）** — 只碰你必须改的：
- 不改相邻代码、注释、格式（哪怕你觉得不好）
- 不重构没坏的东西
- 匹配已有代码风格（哪怕不是你习惯的）
- 发现无关的死代码 → 提醒用户，不要自作主张删
- 底线：**每行改动都能追溯到用户的需求**

### Phase 3: 自检

回答这 3 个问题（不需要输出给我看，自己心里过一遍）：

1. **"有没有我应该读/搜但跳过了的东西？"** → 有的话去补
2. **"每个 Todo 的结果我都实际验证了吗？"** → 没验证的回去验证
3. **"如果现在用户仔细检查我的输出，会发现遗漏吗？"** → 会的话先修

**3 个全 Yes → 可以结束。任一个不确定 → 先补上。**

**结束时额外检查**：生成的文档是否已按 CLAUDE.md 规则归档到工作文件夹？

## 长会话专项

会话超过 15 轮后，每做 5 个 Todo 检查一次：
- 输出是否变短了？
- 工具调用是否比之前少了？
- 是否开始凭记忆而非重新读取？

**任一出现 → 切换到精简模式，重新规划剩余任务。**

## 效率提醒

| 信号 | 意思是你在走弯路，停下来 |
|------|------------------------|
| "大概就是这样" | 不精确。去确认。 |
| "这个先跳过后面补" | 后面不会补。现在做。 |
| "前面读过应该没变" | 重读只需 5 秒。重读。 |
| 工具调用成功但没看输出 | 调用白调了。看输出。 |
| Todo 标记完成但没验证 | 撤回来。验证。 |

## 本 SKILL 对自身同样生效

如果连这套规则都懒得执行——恰恰说明你现在最需要它。别骗自己。

**过程纪律 + 代码质量，缺一不可。** 流程对但代码冗余 = 返工。代码简洁但流程跳步 = 遗漏。两把刀都得快。
